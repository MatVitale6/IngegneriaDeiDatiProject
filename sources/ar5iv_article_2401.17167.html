Abstract

The recent trend of using Large Language Models (LLMs) as intelligent agents in real-world applications underscores the necessity for comprehensive evaluations of their capabilities, particularly in complex scenarios involving planning, creating, and using tools. However, existing benchmarks typically focus on simple synthesized queries that do not reflect real-world complexity, thereby offering limited perspectives in evaluating tool utilization. To address this issue, we present UltraTool , a novel benchmark designed to improve and evaluate LLMs‚Äô ability in tool utilization within real-world scenarios. UltraTool focuses on the entire process of using tools - from planning and creating to applying them in complex tasks. It emphasizes real-world complexities, demanding accurate, multi-step planning for effective problem-solving. A key feature of UltraTool is its independent evaluation of planning with natural language, which happens before tool usage and simplifies the task solving by mapping out the intermediate steps. Thus, unlike previous work, it eliminates the restriction of pre-defined toolset during planning. Through extensive experiments on various LLMs, we offer novel insights into the evaluation of capabilities of LLMs in tool utilization, thereby contributing a fresh perspective to this rapidly evolving field. The benchmark is publicly available at https://github.com/JoeYing1019/UltraTool .

1 Introduction

Recent advancements in equipping Large Language Models (LLMs) ( du2022glm ; touvron2023llama ; vicuna2023 ; bai2023qwen ) with external tools ( patil2023gorilla ; schick2023toolformer ; qin2023tool ) have markedly improved capability of AI systems in solving complex real-world tasks. As this field evolves, it becomes crucial to conduct a comprehensive evaluation covering full aspects of tool utilization, particularly within complex real-world contexts. As depicted in Figure 1 (a), addressing real-world tasks often necessitates not only the planning and usage of multiple tools but also the creation of new tools, when the existing tools are not enough to meet all specific requirements. However, existing benchmarks often focus only on limited dimensions of this entire process. Besides, the user queries in most existing benchmarks exhibit limitations in mirroring the complexity of real-world tasks and unreasonable dependency on pre-defined toolsets.

To tackle these challenges, we introduce UltraTool , a benchmark covering a wide range of evaluation dimensions. It is constructed on complex, real-world queries and involves evaluation of tool-independent natural-language planning and advanced tool creation capabilities. UltraTool comprises 5,824 examples, spanning 22 diverse domains and incorporating 2,032 tools.

UltraTool comprehensively evaluates the tool utilization process including six dimensions covering three aspects: ‚ë† Planning : decompose the complex goal into logical sequence of simpler sub-tasks. This is essential for effective problem-solving. Tool Creation comprises two dimensions: ‚ë° awareness - assessing whether existing tools suffice, and ‚ë¢ creation - developing the necessary tools if existing ones are inadequate. Tool Usage involves three dimensions: ‚ë£ awareness - determining which sub-tasks require tools, ‚ë§ selection - choosing the appropriate tools, and ‚ë• usage - specifying the input parameters for these tools. As demonstrated in Table 1 , recent benchmarks in tool utilization tend to focus only on certain dimensions. For examples, tang2023toolalpaca ; xu2023tool ; patil2023gorilla ; li-etal-2023-api ; qin2023toolllm propose benchmarks for evaluating tool usage (‚ë•). huang2023metatool propose a benchmark to evaluate LLMs‚Äô tool usage awareness (‚ë£) and tool selection ability (‚ë§). Comparing with them, we further evaluate the model capability in planning (‚ë†) and tool creation (‚ë° and ‚ë¢), which is crucial for LLMs to adeptly navigate complex and varied user demands.

Moreover, the queries in UltraTool have better realism and complexity compared to existing benchmarks. The natural workflow of tool-augmented task solving includes: ‚Äúuser queries LLMs ‚Üí ‚Üí \rightarrow LLMs pose a solution plan ‚Üí ‚Üí \rightarrow LLMs create/select tools to solve sub-task within the plan‚Äù. Importantly, the query and the plan should not be constrained by pre-existing tools. In contrast, existing benchmarks ( qin2023toolllm ; li-etal-2023-api ; huang2023metatool ) typically constructed starts with collecting toolset, generates simulated queries with tools randomly selected from the toolset, and then annotates the solution as shown in Figure 1 (b). Despite the collecting efficiency, the generated queries may not accurately mirror real user demands. Furthermore, the randomly selected tools may lack a coherent logical relationship, potentially leading to skewed influences on the subsequent process. To ensure the complexity and reality of queries, we collect real-world multi-domain user queries with high complexity. Specifically, we engage experts from various domains to craft complex queries that reflect real-world needs and potentially incorporate the use of diverse tools. We also ask these experts to list potential tools as the initial toolset, then an automatic toolset completion process is followed to generate any missing tools, ensuring that every potential tool-related need in the user query can be addressed. Finally, we match the tools from the completed toolset to guarantee a natural and smooth logical flow when multiple tools are used.

A distinct feature of UltraTool is it explicitly evaluates natural language (NL)-based plan, which simplifies task solving by decomposing a complex goal into several simpler sub-tasks described in NL. Each sub-task is then solved by tool creation and usage. Therefore, prior planning overcomes the restriction brought by limited pre-existing tools.

We conduct experiments on several LLMs and make in-depth analyses about strengths and challenges in the tool utilization of LLMs. Contributions of this work can be summarized as follows: (1) UltraTool is a comprehensive evaluation benchmark derived from complex real-world queries, covering six key dimensions for tool utilization. (2) UltraTool explicitly evaluates NL-based planning and advanced tool creation capabilities. (3) Extensive experiments uncover the limitations and inspire the future direction of LLMs in tool utilization.

2.2 Query Collection

To ensure comprehensive coverage across a variety of domains and to meet the diverse requirements encountered in real-world scenarios, as shown in Table 2 and Figure 4 , we carefully selected more than 20 domains such as Alarms, Trains, Flights, Hotels , among others. In detail, as depicted in Figure 3 (a), we collaborated with a wide array of domain-specific experts to gather complex, de-identified user queries that necessitate complex tool usage. To better align each query with its corresponding tool usage, we also requested that the experts suggest several potential tools for each query. These suggestions are formatted in accordance with the style of the ChatGPT plugins 1 1 1 https://platform.openai.com/docs/guides/function-calling , which aided in crafting an initial toolset for each query. The detailed guidelines provided to the experts for query collection can be found in the Appendix LABEL:sec:query_guideline .

As shown in Figure 3 (a), to generate a more diverse and challenging set of queries, we utilized GPT-4 to generalize and complicate the initial queries. We then merge the original, generalized, and complicated queries into a comprehensive collection. Additionally, every query underwent a manual review to guarantee quality and applicability, and queries that did not align with actual human needs were removed to ensure relevance.

2.3 Solution Annotation

Due to the complexity of the queries addressed by UltraTool , a multi-step planning approach is essential for effective problem-solving. Consequently, UltraTool employs a NL-based multi-step plan as the backbone of the solution, wherein each step includes a corresponding tool calling message. By breaking down user queries into simpler tasks via this planning process, these tasks can subsequently be executed by the designated tools. This strategy not only in line with the problem-solving logical of human but also reduces the complexity of tool usage, thereby facilitating the handling of more challenging tasks.

Furthermore, we explore an automated, high-quality solution annotation methodology utilizing GPT-4, which encompasses: (1) plan annotation; (2) tool creation and plan refinement; (3) tool calling message annotation; and (4) tool merge.

Plan Annotation.

As illustrated in Figure 3 (b), upon receiving a user query, GPT-4 is utilized to formulate an initial multi-step, tree-structured plan [ s 1 , ‚Ä¶ , s n ] subscript ùë† 1 ‚Ä¶ subscript ùë† ùëõ [s_{1},...,s_{n}] . This fundamental plan is deliberately crafted without including a predefined toolset. Such a strategy guarantees ensures a primary focus on task decomposition and the structural integrity of the NL-based plan. Moreover, it adeptly avoids the potential limitations that could surface when the initial toolset not fully meet the requirements of the user query.

Tool Creation and Plan Refinement.

The initial toolset may not meet all the requirements within the complex user queries in UltraTool , and the processes of generalization and complication might lead to the emergence of new tool usage demands. Therefore, as depicted in Figure 3 (b), to accommodate the need for tools not present in the initial toolset, we employ GPT-4 to discern if the creation of new tools is necessary and create the needed new tools in prescribed format.

Subsequently, we implement an automatic refinement process to enhance the quality of the NL-based plan. Given that the existing toolset has already fulfilled all the requirements of the user query, we provide GPT-4 with the query, the initial plan, and the comprehensive toolset (which includes both the initial and the newly created tools). This allows GPT-4 to refine the plan further in terms of the comprehensiveness of information included and the compatibility with tool callings.

Tool Calling Message Annotation.

As illustrated in Figure 3 (b), we utilize GPT-4 to annotate the sequence of tool calling message [ t 1 , ‚Ä¶ , t n ] subscript ùë° 1 ‚Ä¶ subscript ùë° ùëõ [t_{1},...,t_{n}] . In detail, given the query, the refined plan, and the comprehensive toolset, GPT-4 initially differentiates between the tool-free steps and tool-usage steps . Subsequently, for each tool-usage step , GPT-4 selects the most appropriate tool from the comprehensive toolset. Moreover, utilizing the user‚Äôs query and the contextual information within the plan, GPT-4 generates the necessary arguments for the chosen tool.

It is also noteworthy that UltraTool incorporates nested tool callings, wherein the output from one tool serves as a requisite parameter for another, subsequent tool. This feature significantly enhances our benchmark‚Äôs alignment with the complexities encountered in real-world scenarios.

Tool Merge.

Due to each tool being created separately for the queries, there may be similar tools that exist for the whole benchmark. To address this issue, we manually merge groups of tools that have similar functions into a single tool. Specifically, we combine the arguments and results fields of similar tools with removing any duplicate properties, and then we write an appropriate name and description for the merged tool. The remained toolset of UltraTool contains 2,032 distinct tools.

2.4 Manual Refinement

To guarantee and further enhance the data quality within UltraTool , we have engaged in a careful manual refinement process for all samples, as illustrated in Figure 3 (c). This process includes eliminating redundant steps, supplementing missing steps for coherent planning logic, rectifying tool usage demands and inappropriate tool selections, completing and amending the tool calling messages, and deleting data of substandard quality. We employe six experts endowed with specialized knowledge to execute this procedure, and we ensure that all the polished data undergoes a double-check to maintain high quality standards. A comprehensive refinement guideline for the experts is provided in Appendix LABEL:sec:refine_guideline .

3.2 Planning

Given a query Q ùëÑ Q , LLMs need to decompose the user query and generate a step sequence in the form of a hierarchical structure [ p 1 PL , p 2 PL , ‚Ä¶ , p n PL ] superscript subscript ùëù 1 PL superscript subscript ùëù 2 PL ‚Ä¶ superscript subscript ùëù ùëõ PL [p_{1}^{\text{PL}},p_{2}^{\text{PL}},...,p_{n}^{\text{PL}}] , to serve as the plan. We utilize the Multi-Dimensional Point-Wise LLM-as-Judge Method , applied at a global level, to evaluate LLMs‚Äô planning ability. The generated plan is evaluated across six designed dimensions:

(1) Accuracy: The generated plan must be aligned with the user query‚Äôs objectives.

(2) Completeness: The plan should encompass all tasks and constraints mentioned in the user query, ensuring no elements are omitted.

(3) Executability: Every step in the generated plan ought to be logical and executable, forming a sequence that enables the progressive fulfillment of the user‚Äôs request.

(4) Syntactic Soundness: The language of the generated plan should be grammatically sound.

(5) Structural Rationality: The plan should exhibit a well-organized, tree-like hierarchical structure.

(6) Efficiency: The plan must be concise and efficient, avoiding unnecessary complexity.
