\method : Multi-agent Task Planning using Classical Planners and Large Language Models

Ishika Singh 1 , David Traum 1 , Jesse Thomason 1 1 Computer Science Department, University of Southern California {ishikasi, traum, jessetho}@usc.edu
Abstract

Classical planning formulations like the Planning Domain Definition Language (PDDL) admit action sequences guaranteed to achieve a goal state given an initial state if any are possible. However, reasoning problems defined in PDDL do not capture temporal aspects of action taking, for example that two agents in the domain can execute an action simultaneously if postconditions of each do not interfere with preconditions of the other. A human expert can decompose a goal into largely independent constituent parts and assign each agent to one of these subgoals to take advantage of simultaneous actions for faster execution of plan steps, each using only single agent planning. By contrast, large language models (LLMs) used for directly inferring plan steps do not guarantee execution success, but do leverage commonsense reasoning to assemble action sequences. We combine the strengths of classical planning and LLMs by approximating human intuitions for two-agent planning goal decomposition. We demonstrate that LLM-based goal decomposition leads to faster planning times than solving multi-agent PDDL problems directly while simultaneously achieving fewer plan execution steps than a single agent plan alone and preserving execution success. Additionally, we find that LLM-based approximations of subgoals can achieve similar multi-agent execution steps than those specified by human experts. I Introduction

Symbolic planning problems specified in PDDL tend to explore single agent plans, and changing these domain specifications to enable planning with multiple agents requires a human expert. Unfortunately, multi-agent planning introduces an exponential growth in the search space over possible plans, since each agent can take an action at each timestep. In this paper, we explore whether the commonsense reasoning abilities in LLMs can take advantage of the lexical semantics encoded in the variable, action, and other ontological names of an expert-written planning domain to predict subgoals for individual agents that, when executed together, will achieve a given global goal. We consider both purely symbolic execution domains where transition functions between world states are deterministic and hairier simulation execution domains that introduce stochasticity in environment states and transitions.

Classical planning. Classical or automated task planning algorithms have been widely applied in autonomous spacecrafts, military logistics, manufacturing, games, and robotics. The automated STRIPS planner, for example, operated the Shakey robot [ 1 ] in 1970. Classical planning algorithms classically work with finite, deterministic, and full state information, admitting guaranteed plan generation when a path from the initial to goal states is possible. Some other frameworks were also shown to be useful for robot planning [ 2 , 3 ] . Planning domain description language (PDDL) and answer set programming (ASP) are popular specification formats for planning domains [ 4 , 5 , 6 , 7 ] .

Planning with LLMs. Several works have shown that LLMs can guide agent actions by acting as planners [ 8 , 9 ] , but such stochastic, generative approaches lose the success guarantees of classical planners. Another line of work shows that LLMs plan better with API-based programmatic plan generation [ 10 , 11 , 12 ] , which introduces some symbolic structure and constraints but still does not ensure success. Program synthesis for planning has been previously proposed in LEAPS [ 13 ] , which generates programs using a learned latent space. In recent developments [ 14 , 15 ] , PDDL has been used as a translation medium, with LLMs used to generate either a PDDL plan or goal. Subsequently, a classical planner is used to plan for the PDDL goal. Generating a PDDL goal eliminates the need to track execution state, required when generating a plan using LLMs instead. However, needing to use a classical planner necessitates specification of full domain. LLMs have also been used for multi-agent planning through code generation [ 16 ] and task allocation through dialogue and path planning [ 17 ] . In this paper, we use LLMs to infer commonsense ways to decompose a goal, but leave formal planning to a PDDL system to ensure plan correctness.

II-B Planning Domain Definition Language (PDDL)

Planning Domain Definition Language (PDDL) serves as a standardized encoding language to represent planning problems. In this paper, we consider deterministic, fully-observable planning problems. The planning problem in PDDL is represented by two files: domain and problem. The domain file defines the environment in terms of the object types and predicates (Figure 1(a) ), followed by action and their parameters, required preconditions, and postcondition effects (Figure 1(b) ). Some conditions relate to the agent (holding ?x) , and others to the environment (on-table ?x) . The problem file defines an instance of the domain with a list of objects, initial state and goal conditions (Figure 1(c) ). We use a PDDL planner that takes in a domain and problem file and searches for action sequences leading from the initial state to a goal state, heuristically returning the shortest such plan found, if any (Figure 1(d) ).

IV Experiment setup

Our principal experiments measure whether \method can lead to more efficient two-agent tasks than a solution to a single-agent PDDL problem in terms of agent actions while simultaneously requiring less planning time than needed to solve a two-agent PDDL problem formulation. Overall, we find that \method achieves that balance when both: the planning domain considers agent state in action preconditions; and the planning problem to solve contains two or more partially independent subgoals rather than requiring strict action sequencing.

IV-A Domains

We evaluate \method on 5 symbolic domains utilized in prior work that leveraged LLMs for planning [ 15 ] and in one simulation environment [ 18 ] .

IV-A 1 Symbolic Domains

Each symbolic environment defines 20 tasks that vary in number of the environment objects and optimal plan length. These domains are frequently used in classical planning competitions. 1) BARMAN : A bartender with 2 hands has been assigned the job of preparing cocktails for a customerâ€™s order, using the specified ingredient and appropriate tools, and striving to accomplish this with the fewest possible actions. 2) BLOCKSWORLD : A collection of block piles arranged on a table, and a robot is tasked to reorganize them into a specified configuration while adhering to the simple physics principles. 3) TERMES : The objective of a robot is to construct complex structures by transporting and positioning blocks, as well as using them as a means to ascend and move adjacent blocks. 4) TYREWORLD : The robot is assigned with changing flat tires, which involves tasks such as removing flat tires, inflating the intact tires, tightening nuts, and returning tools to the boot, all in the correct sequence. This domain does not have a agent state while taking an action, for example, the agent need not be holding a tool first in order to apply a tool to a tire. 5) GRIPPERS : A set of robots with 2 grippers each are given a task to move objects among different rooms.

IV-A 2 Simulated Domain

We use AI2THOR simulator for multi-agent task execution. We build on the textworld abstraction of AI2THOR , AlfWorld , that enables high-level planning while abstracting low-level execution. It provides the domain and tasks encoded in PDDL for automated task plan generation via classical (e.g. PDDL) planners. We extend the support to multi-agent PDDL planning, by initializing multiple agents in the PDDL initial environment state as well as the simulator. The multi-agent simulation, AI2THOR , is implemented asynchronously, i.e., each agent gets to take an action at every time step, however, due to the underlying AI2THOR support in each simulator timestep only one agent can be issued a command like move forward/pickup/etc. We build a parallel plan execution framework that allows action queuing if an agent is occupied with a previous action, such as â€˜go to microwave 1â€™ which involves multiple navigation steps. The oracle multi-agent navigation pipeline uses object location and position and other agentâ€™s position to navigate. Reachable positions, where an agent can navigate to, at any time change when another agent moves in the environment as it occupies space, akin, to any other object. The simulation supports path replanning around another agent while navigating, however, there are scenarios where the agents are deadlocked together if thereâ€™s no way around the collision. At this point, we defer a randomly chosen agent to a randomly chosen location to relieve from the deadlock. Thereafter, the agents continue to execute their queued actions.

IV-B Approaches Considered

We compare \method against single agent and multi-agent PDDL planning for all the above domains. To enable multi-agent planning for symbolic domains, we edited PDDL domain and problem files to reflect multiple agents. We added â€˜agentâ€™ parameter to all the actions and agent-specific predicates in the domain, and initialized 2 agents along with their initial states in the prolem files of 20 tasks. This extension requires understanding of PDDL functioning. Moreover, initializing another agent increases the number of possible actions at any state as well introduces addition state conditions, leading to larger search space. We also define 4 long-horizon tasks and their corresponding PDDL problem files for our evaluations in the simulated domain, as shown in Table III . Some objects may also be inside another containers like drawers/cabinetes as per the PDDL initial state specification, making the resultant plans longer. We will open-source our defined domain and problem files. We consider the following approaches for the above domains,

1) SA PDDL : PDDL planning in the above domains for a single agent. This baseline will compare \method with a single agent completing the task using a single agent plan generated by the PDDL planner.

2) MA PDDL : PDDL planning in the above domains for two agents. This baseline will compare \method with multiple agents completing the task using a multi-agent plan generated by the PDDL planner.

3) \method : multi-agent plan with an LLM-guided helper and main agents - our method as shown in Figure 2 . This method compares LLM-based division of the task into a partially independent subgoal, executed in parallel, when planning for the two agents are done separately as single agent PDDL problems.

IV-C Evaluation metrics

We measure planning time and plan execution length to asses the efficacy of plans generated by different approaches described above. A method that achieves lower planning time and a shorter execution length is desirable as it would be faster in finding a plan and executing it. Note that we do not report task success metric, since classical planning methods are always successful. Moreover, \method ensures success as the system defaults to a single agent if the inferred subgoal is faulty.

1) Planning Time: total time taken by the PDDL planner to explore the full state space, capped by a time limit. For \method , planning time is the sum of helper PDDL planning time, main PDDL planning time, and LLM inference time. We report LLM inference time separately as an overall average instead of adding each taskâ€™s inference time to average out any LLM API call delays.

2) Execution Length: For multi-agent PDDL and \method , it refers to the effective length of the plan when multiple agent plans are executed in parallel. For single agent, execution lenth is equal to best PDDL plan cost, however, for multi-agent, execution lenth â‰¤ \leq best PDDL plan cost. For symbolic domains, we utilize Algorithm 1 to find the best possible execution length, by dynamically finding the best parallel plan execution strategy. For simulated domain, execution length refers to number of environment steps taken, where all agents take a step sequentially in each environment step, including navigation and interaction steps. Depending on action preconditions and execution state, we also consider some agentâ€™s action at a time step to be a â€˜waitâ€™ or a â€˜no operationâ€™ action.

VI Conclusion

We propose \method , a method to decompose a single agent planning problem into a two-agent planning problem in several symbolic domains and one simulated domain. \method leverages commonsense from LLM to effectively divide a problem between two agents for faster execution, while also preserving execution success using classical planning guarantees. Our results show that LLM-based goal decomposition leads to faster planning time than the multi-agent PDDL problem and shorter plan execution steps than the single agent execution. We additionally show that LLM-inferred subgoals in \method approximate those specified by a human expert.
