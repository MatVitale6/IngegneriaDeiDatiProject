Abstract

How-to procedures, such as how to plant a garden, are ubiquitous. But one size does not fit all - humans often need to customize these procedural plans according to their specific needs, e.g., planting a garden without pesticides. While large language models ( LLM s) can fluently generate generic procedures, we present the first study on how well LLM s can customize open-domain procedures. We introduce CustomPlans , a probe dataset of customization hints that encodes diverse user needs for open-domain How-to procedures. Using LLM s as CustomizationAgent and ExecutionAgent in different settings, we establish their abilities to perform open-domain procedure customization. Human evaluation shows that using these agents in a Sequential setting is the best, but they are good enough only ~51% of the time. Error analysis shows that LLM s do not sufficiently address user customization needs in their generated procedures.

1 Introduction

How-to procedures (or procedural plans) serve as building blocks in decision-making processes, even making up for a large fraction of search engine queries 1 1 1 https://observer.com/2018/12/google-most-searched-2018-report-top-questions/ . WikiHow contains a large collection of high-quality, expert-authored How-to procedures about diverse topics. There are a myriad of ways to tailor these generic How-to procedures to suit users by using different customization hints. For example, for a How-to procedure like ‚ÄúHow to stay healthy‚Äù, users can prefer yoga, or a Mediterranean lifestyle, or be dealing with an injury, or more. It is infeasible to manually compile innumerable customizations of a procedural plan, and thus it is important to build an automated system to do this.

Customization hints implicitly encode multiple requirements and are subjective ‚Äî for example, a Mediterranean lifestyle can involve a diet of whole grains, using healthy fats like olive oil, the occasional glass of wine, walking, and more. These implicit customizations are best expressed in natural language since it is more practical in a real-world, open-ended setting. Thus, we need an automated method that can understand these implicitly expressed needs and make consistent revisions to customize how-to procedures.

Prior work has focused on building supervised customization models for limited domains such as shopping, news and video games Rashid et¬†al. ( 2002 ); Togelius et¬†al. ( 2007 ) . Majumder et¬†al. ( 2019 ) focus on just customizing cooking recipes while Antognini et¬†al. ( 2023 ) are limited to certain aspects of customization. An open-domain setting with natural language customization hints requires a supervision-free model such as LLM s.

This paper presents the first study of using LLM s to customize open-domain How-to procedures. First, we compile a new dataset, CustomPlans , that contains diverse procedures and customization hints. We manually collect a small, high-quality dataset with diverse procedures and customization hints and also re-formulate two existing WikiHow based datasets ‚Äî Choice-75 Hou et¬†al. ( 2023 ) and CoPlan Brahman et¬†al. ( 2023 ) ‚Äî that contain user scenarios which can serve as customization hints.

We then evaluate how well LLM s can customize procedures. We conduct a set of experiments inspired by prior work on building supervised models that can edit procedures according to constraints. Through human evaluation, we establish that using first customizing the original procedure for a goal and then fixing issues related to execution is the best method to generate procedures that satisfy customization hints. Even so, we find that LLM s fail to sufficiently address the customization needs, often generating procedures that miss important steps.

In summary, our contributions are:

  • ‚Ä¢

    CustomPlans , a probe dataset to study how well LLM s can customize open-domain procedures when provided hints from users.

  • ‚Ä¢

    Using a multi-agent framework, LLM s are able to generate executable customized procedures ¬†51% of the time, leaving significant room for improvement.

  • ‚Ä¢

    Models are unable to fully address the implicit requirements of user customization hints in their generated procedures.

2 The CustomPlans dataset

Prior work on editing and customizing procedures has focused on specific domains. So, data for the task of procedure customization is limited. To alleviate this problem, we re-purpose two related datasets and collect more high-quality data from WikiHow to create the CustomPlans probe dataset.

2.1 CustomPlansReal

Each WikiHow article consists of a goal (indicated in the title starting with ‚ÄúHow to‚Äù) and ordered, richly described steps that lead to that goal. It already contains articles that include multiple methods for a task. Additionally, search queries can be used to retrieve and select articles broadly related to the same goal. However, user interaction signals like fine-grained querying related to specific versions of procedures are not available publicly. Such signals can be used to improve user experiences. Therefore, we create a new probing set that contains a procedural goal and an individual user‚Äôs relevant context.

To compile this data, we start by identifying 9 diverse domains. For each domain, we employ two methods to curate WikiHow articles. 1) Search Results: We devise relevant, popular and pragmatic goals which have relevant instructions present in WikiHow. We present each goal to users of relevant persona and ask them to provide their feedback according to their personal preferences. For example, for the broad goal of staying healthy, a collected user persona was that of a teenager looking to build long-term healthy habits. 2) Multiple Methods: For the same set of broad goals, we filter out WikiHow articles that contain multiple methods to achieve the same goal. Given the broad goal, users are required to provide situational feedback corresponding to each method. Using this approach, we collect 106 customized goals (corresponding to user feedback) over 9 domains and build CustomPlansReal , which contains customization hints collected from humans.

2.2 CustomPlansSimulated

To study decision branching in scripts, Hou et¬†al. ( 2023 ) created Choice-75 , a benchmark of 565 data points which requires selecting the next step in a procedure given descriptive scenarios. These scenarios are valid only at a step level. However, they can also be incorporated into the full procedure if treated as constraints. We treat these scenarios as customization hints.

CoPlan is a dataset of machine-generated, human-verified scripts. We use the test set of 861 data points for the counterfactual plan revision task for our purposes. Brahman et¬†al. ( 2023 ) collect goals from diverse topics and prompt text-curie-001 to generate a set of ordered steps as a plan to achieve that goal. text-curie-001 is also used, in a few-shot manner, to generate conditions that can alter these plans. Human verification of these components leads to the creation of a test set that contains a goal, an uncustomized procedure and a relevant condition. These conditions describe prerequisites to be fulfilled that require the original procedure to be customized in a specific manner. We treat these conditions as customization hints. The combination of data from these two sources is referred to as CustomPlansSimulated since it contains customization hints generated automatically.

2.3 Statistics: CustomPlans dataset

We randomly select 100 procedures and their corresponding customization hints from CustomPlansSimulated and add them to CustomPlansReal to form the CustomPlans evaluation set. Overall, CustomPlans contains 206 data points, each of which is made up of a goal, a list of steps to achieve that goal and a customization hint from a user according to which the procedure should be modified. It contains 106 unique goals and 203 unique customization hints. Table¬†1 shows examples from the dataset.

We also store relevant metadata for each user persona providing the annotation such as their level of expertise and their constraints like dietary preferences or availability of tools. 1.2% of these customized procedures can only be performed by domain experts, 7.6% need an intermediate level of expertise, and the rest are beginner-friendly. The user comments contain implicit or explicit constraints ‚Äî 68% contain hard prerequisites, 30% reflect some user preference and the rest mention a target refinement to be achieved. 3 Experiment Details

3.1 Task

The task, illustrated in Figure¬†3 , involves taking as input the goal G ùê∫ G of the procedure P ùëÉ P as well as a user customization hint H ùêª H . The output should be a procedure to fulfill the goal while also taking into account the user scenario described in the hint P ‚Ä≤ superscript ùëÉ ‚Ä≤ P^{\prime} . To tailor procedures according to customization hints, one needs to make implicit inferences out of the hints, identify the steps that require changes, and finally consistently apply changes to the different steps. We introduce a multi-agent prompting framework to establish the abilities of state-of-the-art LLM s to customize procedures according to user feedback using the CustomPlans dataset. Through our experiments, we aim to understand how well models can modify generic procedures to incorporate aspects provided in customization hints.

3.2 Models

Recently, model-based agents have been used to perform different types of reasoning in service of achieving a larger goal Yoran et¬†al. ( 2023 ) . To that end, we disentangle the task of generating customized procedures, delegating customization to CustomizationAgent and ensuring the executability of the generated procedure to ExecutionAgent . However, it is unclear how these agents can interact with each other to best generate customized procedures.

Each agent produces a bag of semi-structured edits E ùê∏ E that indicate the operation to be performed (they are only allowed to perform the insert or replace operation), the step in the original procedure P ùëÉ P to perform it on/after, and the new step to replace it with, or insert after it. We then apply these edits deterministically on P ùëÉ P to obtain P ‚Ä≤ superscript ùëÉ ‚Ä≤ P^{\prime} i.e., we find the step number in P ùëÉ P that corresponds to a suggested edit and insert/replace it with the edited text. We experiment with three settings for the defined agents: Uniform , Sequential , and Parallel .

Uniform - We first define a single agent that performs the task of suggesting edits about customization as well as executability on P ùëÉ P to generate P c subscript ùëÉ ùëê P_{c} . This agent is required to understand how to perform both customization towards a hint H ùêª H as well as execution to achieve the goal G ùê∫ G .

Sequential - In this setting, we first obtain a set of edits E c subscript ùê∏ ùëê E_{c} from CustomizationAgent , and apply E c subscript ùê∏ ùëê E_{c} to obtain P c subscript ùëÉ ùëê P_{c} . Next, we use P c subscript ùëÉ ùëê P_{c} to extract another set of edits E e subscript ùê∏ ùëí E_{e} from ExecutionAgent . Finally, we deterministically apply E e subscript ùê∏ ùëí E_{e} on P c subscript ùëÉ ùëê P_{c} to obtain the resulting procedure P ‚Ä≤ superscript ùëÉ ‚Ä≤ P^{\prime} . Here, the agents are used in a sequential order, first suggesting edits to meet customization requirements and next to address any execution-related issues that may arise from those edits.

Parallel - Both CustomizationAgent and ExecutionAgent propose their set of edits, E c subscript ùê∏ ùëê E_{c} and E e subscript ùê∏ ùëí E_{e} , upon P ùëÉ P . However, this gives rise to conflicting edits according to different objectives, both of which are important to generate P ‚Ä≤ superscript ùëÉ ‚Ä≤ P^{\prime} . To address this, we use ResolutionAgent , which takes as input two bags of edits, E c subscript ùê∏ ùëê E_{c} and E e subscript ùê∏ ùëí E_{e} , and produces a merged set of edits E ùê∏ E . Finally, applying E ùê∏ E on the original procedure P ùëÉ P results in P ‚Ä≤ superscript ùëÉ ‚Ä≤ P^{\prime} .

3.3 Evaluation

For open-ended text generation tasks like procedure customization, the absence of an automatic evaluation that correlates well with human judgments is a major challenge Chen et¬†al. ( 2019 ); Ma et¬†al. ( 2019 ); Caglayan et¬†al. ( 2020 ); Howcroft et¬†al. ( 2020 ); Lal et¬†al. ( 2021 ) .

We conduct a human evaluation with a standardized interface to compare different models. We want to measure whether a model produces procedures that meet the requirements of the customization hint, as well as whether the generated procedures can be executed to achieve the goal. To this end, we pose questions about whether a procedure can be performed as is to accomplish the given goal ( Executable ), and whether it satisfies a presented customization constraint ( Customized ). This serves as the human evaluation interface for our task. A sample HIT can be seen in Figure¬†5 .

For each goal, customization hint, and a model‚Äôs generated procedure, we ask 3 distinct annotators to provide judgments about customizability and executability of that procedure. A procedure can be complete in an aspect, missing steps, have extra steps, have underspecified steps, or have incorrect steps. We take majority vote of annotator judgments for the Customized and Executable criteria. If a plan is judged to be both Customized and Executable by a majority of annotators, we consider it to be FullyCorrect .

We report majority statistics for customizability, executability and correctness of different models in Table¬†2 .

Using CustomizationAgent and ExecutionAgent in Sequential order is the best at producing customized procedures P ‚Ä≤ superscript ùëÉ ‚Ä≤ P^{\prime} . It performs better than using just one agent in Uniform , which, in turn, is better than using both agents in Parallel setting. We note that execution accuracy for all models are similar. However, performance differs based on how well a model can address implicit and explicit customization needs.

Procedures are not sufficiently customized . All methods suffer from the problem of missing steps, indicating that none of them adequately address the requirements stated implicitly or explicitly in the customization hint for the procedure.

Procedures are not direct . For the executability aspect, each method suffers the most from generating unnecessary steps, which often hinder achieving the goal of the procedure G ùê∫ G . Furthermore, procedures generated by Parallel also suffer from missing steps. We hypothesize that this problems arises because ResolutionAgent is unable to correctly merge bags of edits from CustomizationAgent and ExecutionAgent .

5 Related Work

Over the past few years, as information about users has been readily available, user-facing technology has become increasingly customized. Customization has been widely studied in the context of such technology, like search engines Rashid et¬†al. ( 2002 ) , chatbots Majumder et¬†al. ( 2020 ) and game content Shaker et¬†al. ( 2010 ); Togelius et¬†al. ( 2007 ) . There is some work on customized procedural content generation Togelius et¬†al. ( 2011 ); Yannakakis and Togelius ( 2011 ) . However, they focus on video game content rather than everyday procedures related to pragmatic goals. Previous research has shown the value of customization in various settings. Kapusta et¬†al. ( 2019 ) present a method to customize collaborative plans for robot-assisted dressing. Building customized care plans is crucial for patients with severe illnesses Lin et¬†al. ( 2017 ); Anbari et¬†al. ( 2020 ) .

Procedural text understanding addresses the task of tracking entity states throughout the text Bosselut et¬†al. ( 2018 ); Henaff et¬†al. ( 2017 ) . Generating such texts Aouladomar and Saint-Dizier ( 2005 ) involves different types of understanding such as temporal reasoning and entity linking. Mori et¬†al. ( 2014 ) generated a procedural text from a meaning representation taking intermediate states into account. ChattyChef Le et¬†al. ( 2023 ) uses the conversational setting to generate cooking instructions and iteratively refine its step ordering. To study decision branching in procedural text, Hou et¬†al. ( 2023 ) generated user scenarios and choices for various steps in a procedure and presented Choice-75 . For the task of counterfactual planning, CoPlan Brahman et¬†al. ( 2023 ) collects conditions and combines them with a revised list of steps. Majumder et¬†al. ( 2019 ) use historical user preferences to generate customized recipes. But they only focus on one domain (here, cooking) and only on one dimension to model customization. LLMs have been shown to generate procedural text well but it is unclear how they can be customized for diverse user preferences.

Customized language models are designed and trained to capture language patters specific to individual users. Such models are often better than using more general models for human-level NLP tasks Lynn et¬†al. ( 2017 ) . King and Cook ( 2020 ) examined methods for creating customized language models using interpolating, fine-tuning and priming. Similarly, Welch et¬†al. ( 2022 ) present another approach for fine-tuning and interpolation to customize LMs. In the context of LLMs, role-based prompting is a recent trend. Pseudo Dialog Prompting Han et¬†al. ( 2022 ) is a method to build prompts for LLMs so that chatbots mimic fictional characters, while Vincent et¬†al. ( 2023 ) release a dataset of character annotations to induce personas in LLM output. However, it is still unclear how to encapsulate more fine-grained aspects of customization.

6 Conclusion

This paper studies the capabilities of current LLM s to customize open-domain How-to procedures. We introduce a probe dataset, CustomPlans , for this task. Using LLM s as CustomizationAgent and ExecutionAgent in different settings, we establish their abilities to perform open-domain procedure customization. Human evaluation shows that using these agents in a Sequential setting is the best, but they are good enough only ~51% of the time. Error analysis shows that LLM s do not sufficiently address user customization needs in their generated procedures. Further, the generated plans contain extra steps because of which they cannot be often executed.
